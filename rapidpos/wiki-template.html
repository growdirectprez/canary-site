T — Transaction Pipeline

The data foundation module. It connects the NCR Counterpoint REST API to the CRDM in real time, normalizing every transaction — sales, returns, holds, markdowns — from every terminal into canary_sales.

STATUSLive · Phase 1 polling OWNERGrowDirect Platform DEPENDS ONCounterpoint REST · canary_sales UPDATED2026-04-22

T — Transaction Pipeline is the data foundation module. Every other Canary module depends on T being live and current. If T is down, canary_sales goes stale; if canary_sales is stale, every downstream tool — inventory, loyalty, pricing — returns yesterday's truth. This page is the operator's contract for keeping it honest.

Key insight

T is the only module that runs without the MCP Server. It is upstream of the protocol layer entirely — its job is to make the CRDM continuously match Counterpoint, so that every tool call answered by the MCP Server is answered against fresh data.

§ 01Two phases — polling, then push

T ships in two delivery phases. Phase 1 is live for every retailer on the platform; Phase 2 rolls out per-tenant once the Counterpoint webhook surface is whitelisted with the retailer's IT team.

Phase 01 · Polling

60-second cycle

Every Counterpoint terminal's transaction log is pulled on the cycle, diffed against the last snapshot, and written to canary_sales. Idempotent — safe to replay.

Latency ≤ 60s
───►
Phase 02 · Webhook push

Sub-second propagation

Counterpoint fires an event on every transaction. T receives it, validates it, writes to canary_sales, and invalidates Redis cache for affected SKUs.

Latency < 500ms

The transition from Phase 1 to Phase 2 is per-retailer and reversible. If the webhook channel falters, T falls back to the 60-second poll without operator intervention. Failure modes are documented in the integration guide.

§ 02Schema — every row in canary_sales

Every transaction row in canary_sales is canonical. The shape is the same whether the row arrived via polling or webhook — Phase 2 is a latency upgrade, not a schema change. canary_sales schema reference is the source of truth; the columns below are the operator-relevant subset.

The polling cycle pulls a batch every 60 seconds; the example below is one row of that batch, after normalization, ready for write.

canary_sales · INSERT payload

{
  "store_id": "PAS",
  "terminal_id": "REG-04",
  "transaction_type": "SALE",
  "sku": "AGAVE-BLU-5G",
  "quantity": 1,
  "unit_price": 39.99,
  "customer_id": "a3f1c2e0-8d27-4ebb-b1f5-09a7b4d9c1a3",
  "counterpoint_transaction_id": "CP-2026-1147-88419",

  "ingested_at": "2026-04-26T18:14:48Z"
}

Pipeline latency for this row: 26s. Phase 2 push would have logged < 500ms for the same write.

§ 04Operating notes

Idempotency

The counterpoint_transaction_id column is unique. Re-runs of the polling job — manual or automatic — are safe. Inserts collide on the unique key and are dropped silently.

Backfill

To replay history: invoke the tsp:backfill CLI with a date range. The job paginates the Counterpoint /transactions endpoint at 1,000 rows per page, writes via the same idempotent insert path, and emits progress to the operator log. Backfill does not invalidate Redis — cache is rebuilt by the next live event.

Watch · cache drift

If T is paused for > 5 minutes, downstream inventory_lookup calls will surface the cache_age_seconds field accurately, but loyalty point accruals can drift. Re-enable T before issuing high-trust transactional tool calls.
