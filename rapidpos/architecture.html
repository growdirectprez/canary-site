Technical Architecture

The CRDM is the foundation. Everything else is a module.

Three-database PostgreSQL architecture connected to a live NCR Counterpoint transaction stream. The MCP Server sits on top. The agents sit on top of that.

The CRDM

One canonical data model. Three databases.

The Centralized Retail Data Model is a three-database PostgreSQL architecture that normalizes every transaction from every Counterpoint terminal into a unified canonical data model.

It is the layer that neither NCR Counterpoint nor RapidPOS provides — and what all 13 Canary modules depend on.

Transaction Streaming Pipeline

From Counterpoint to canonical, in two phases.

Phase 1 polls the NCR Counterpoint REST API on a 60-second cycle — every sale, return, hold, and markdown from every terminal flows into canary_sales. Phase 2 upgrades to webhook push — inventory events fire on transaction, cache invalidated in real time. The CRDM goes from 60-second to sub-second accuracy.

Phase 1 · Live

60-second polling

Worker calls Counterpoint REST endpoints on a fixed cadence. Idempotent upserts into canary_sales. Redis cache TTL of 60 seconds for reads. Sufficient for inventory accuracy at human-shopping pace.

Phase 2 · Webhook push

Sub-second event-driven

Counterpoint webhook fires on transaction. CRDM ingests, cache key is invalidated, replicas update. Inventory writes propagate to all readers in under a second. Same MCP tool surface — no client changes.

MCP Protocol

JSON-RPC 2.0 over HTTPS + SSE.

The MCP Server exposes tools over JSON-RPC 2.0 via HTTPS with Server-Sent Events for streaming responses. Any MCP-compatible AI client connects without custom integration — Claude, GPT-4o, and any agent built on the Model Context Protocol speaks this natively.

Security model

Credentials stay server-side. Scopes gate every write.

Tenant auth

API key authentication per retailer tenant. Keys are rotatable, scoped to a single Counterpoint instance, and revocable from the RapidPOS console.

Credential isolation

All Counterpoint credentials stay server-side. The MCP client never sees them — only the bearer token issued by the MCP layer.

Rate limiting

Rate limiting is enforced at the MCP layer, not passed through to Counterpoint. Per-tenant quotas; burst allowances configurable per VAR.

Read vs. write

Read-only tools have no state side effects. Transactional tools (holds, payments) require explicit customer consent scoping — transactions:write, customer:read, etc.
