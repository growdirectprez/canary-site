Design Documents

12 SDDs in recommended reading order. Start at 01 if you're new to the codebase.
Source: GrowDirect/docs/sdds/go-handoff/

Orientation — Read First
01
Platform Overview
platform-overview.md
product 		What Canary is and who it serves. Covers the ICP (independent specialty retailer, 1–30 locations), the three accountability rails, the module roadmap, and the security and compliance posture. Narrative layer — read before the architecture SDD.
02
Architecture
architecture.md
platform 		Service topology across 29 modules. Chi HTTP, pgx/v5, Valkey streams, GCP Cloud Run. Shows how the protocol pipeline (Node 1–4), the module stores, and the MCP surface hang together. Start here for the wiring picture.
03
Go Module Layout
go-module-layout.md
conventions required 		Read before creating any file. Defines the single go.mod, package naming, service boundary conventions, port assignments, and cmd/* binary layout. Consistency across 29 services is enforced here.
Data Layer
04
Canonical Data Model
canonical-data-model.md
data active 		ARTS-anchored, SMB-2030 scoped master data contract. Supersedes data-model.md. Nine domains: Item, Customer, Transaction, Inventory, Receiving, Party, Finance, Operations, Device. GSLM provenance from Walmart International. The schema all services write against.
05
Identity
identity.md
auth 		Merchant identity and the tenant model. API key auth for agent clients, federated identity path via GCP Cloud Identity. Covers the company_alias namespace, source_code scoping, and the .jeffe namespace registration flow.
06
Microservice Architecture
microservice-architecture.md
platform 		Inter-service communication contracts, Valkey stream event schemas, error handling conventions, and the audit middleware contract. How 29 services stay consistent without tight coupling.
Protocol Pipeline
07
Webhook Pipeline
webhook-pipeline.md
ingestion live 		Node 2 of the protocol pipeline. HMAC-SHA256 validation, nonce replay protection, payload hash computation, publish to Valkey stream. The front door for all external events (POS, Square, DriftPOS).
08
TSP — Transaction Stream Processor
tsp.md
pipeline live 		Three-subscriber pipeline: Sub1 hash-seal, Sub2 parse-route, Sub3 Merkle-ordinal anchor. Each is an independent consumer group on the Valkey stream. The ordinal anchor is the Evidentiary rail in code.
09
Chirp Detection Engine
chirp.md
detection 		37 detection rules across 10 categories. Rule threshold tuning, alert routing to hawk case management, and the reserved ILDWAC cost-anomaly rule family (Category 11). The signal layer of the Operational rail.
Integration & Billing
10
MCP Service Junctions
mcp-service-junctions.md
mcp live 		26 MCP tools across 7 domain modules (alert, analytics, asset, customer, employee, returns, report). The junction inventory — every event traverses here, which is also where billing and the evidentiary record are created. The agent interface to the platform.
11
DriftPOS Integration Contract
driftpos-integration.md
integration 		CAN-DESIGN-001. Wire protocol between Canary and RapidPOS (DriftPOS). Event types, auth model, delivery guarantees, and the retry/DLQ contract. The primary channel delivery spec for the VAR partnership.
12
Satoshi Cost Rollup
satoshi-cost-rollup.md
billing 		Go types, database schema, calibration loader, metering pipeline, and settlement choreography for satoshi-denominated billing. The Financial rail in code. Every junction traversal pays a toll; the payment IS the audit record.

These 12 SDDs are the recommended reading set for new engineers. The full library in docs/sdds/go-handoff/ contains additional service-level specs (fox, owl, bull, hawk, analytics, receiving, pricing, and others) and operational docs (go-errors, go-observability, go-testing, go-security). Start with the 12 above; reach for the others when you're working in a specific service.
